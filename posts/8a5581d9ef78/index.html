這個檔案定義出了三個prims，並且也定義出了他們的hierarchy，但是可以注意到定義一個prim時有三個單字，例如: def Cube "Box" ，其分別為Specifiers、Schema以及Name。

Specifiers

Specifier總共有三種，分別是 def 、 over 以及 class ，大多數的.usd都可以看見 def ，尤其是越接近下層，也就是流程中越上游的原始檔案，因為這是從無到有define出一個新的prim的時候需要用到的Specifiers。

Specifier class

相對於define出一個實體的prim，也可以用 class 定義一個abstract prim，這個概念類似於coding中的class，你可以定義一個blueprint或者是template。

回到Omniverse可以發現Layer中多了一個 "_box" ，這個”_box”不存在實體，所以你無法從viewport中找到它，同時可以注意到它的attribute size ，也就是Figure 1(a)中Raw USD Properties的 size=4 ，同時，因為prim "Box" 繼承了class "_box" 所以會將預設的 size 覆蓋掉，因此畫面中的Cube變大了。

Figure 1: (a) Class “_box”; (b) Prim “Box”

此時，我們可以嘗試一些有趣的操作，比如把 def Cube "Box" 改成 def "Box" ，結果如Figure 2(a)所示，可以發現 "Box" 純粹就只是 "_box" 的instance，同時viewport中的方塊應該也不見了，因為 "Box" 不再是一個方塊了。

"Box" 此時只是有一個attribute size 的prim。(如果viewport中方塊仍然存在，請嘗試重新讀取整個檔案)

Figure 2: (a) After changing def Cube "Box" to def "Box" ; (b) After changing class "_box" to class Cube "_box"

此時，如果想要將 "Box" ，在viewport中重新顯示成方塊，除了原本的做法 def Cube "Box" 之外，更合理的作法是 class Cube "_box" ，定義類別本身就是Cube的一種，此時因為 "_box" 是Cube，繼承者 "Box" 自然也會是Cube，結果如Figure 2(b)所示。

同時，我們可以在將添加 "Box" 一些local opinion，也就是新的attribute，這個attribute會覆蓋掉 "_box" 的attribute，在OpenUSD中，我們會稱不同.usd中的attribute的value是對應layer的 opinion 。

Specifier over(ride)

但無論是 def 或 class ，他們都會被 over Specifier影響，它的作用有點類似剛剛提到opinion時做的事情，可以在上層使用 over 去override下層的opinion，換言之， over 常常出現在有多個檔案的情況下。

因此，我們使用一樣上篇的 Layout.usda 來舉例，在其中，我們添加這幾行

    over "GroupTransform"

        over "Box" 
        {

之後，用Omniverse打開 Layout.usda 後可以發現 "Box" 的 size 變為10，如Figure 3(a)所示，因為此時 Layout.usda 才是root layer，他的value才是local opionion，而 MyModel.usda 的opinion則會被 over 所覆蓋掉。

Figure 3: (a) After overriding opinion of “ /Geometry/GroupTransform/Box”; (b) After overriding opinion of “/Box”

這邊有一個小地方需要注意的是， Layout.usda 如果省略掉其他的prim只寫

over "Box" 
{
    double size = 10

結果如Figure 3(b)所示，會在外層定義出一個新的Prim，雖然名稱也是 "box" ，但是兩個box的path是不同的，分別是

  • /Geometry/GroupTransform/Box
  • /Box

每個Prim都有一個獨一無二的Path，換言之，只要Path不同，那就是不同的Prim，只要Path相同，就會被視為相同的prim，也就會涉及Compose，例如overriding。

那如果在 Layout.usda 這樣寫

{
    double size = 10
}

結果如Figure 4(a)所示，我們在 Layout.usda 中override class "_box" 的opinion，但在property中會發現 "Box" 的 size 仍然是1，因為這個opinion是來自於繼承的opinion，而 "Box" 的local opinion依舊強於它，就算這個繼承的opinion它是來自上層。

Figure 4: (a) After overriding opinion of class “_box” ; (b) Removing local opinion of prim “Box”

因此，如果想要讓 "Box" 的 size 變為10，就需要把 "Box" 的local opinions拿掉，此時 "Box" 的size就完全由 "_box" 決定，修改後的 MyModel.usda 如下所示，結果如Figure 4(b)。

Schema

概念類似剛剛提到的class，也算一種template，例如 def Cube "Box" 中prim "Box" 的Schema是 Cube ，概念上， "Box" 就是被定義成一個Cube，至於Cube的定義，是事先規範好的常用型態，你完全可以想像成Schema是一個內建的class，你不需要inherits而是直接使用他就好了，這可以說明為什麼可以寫 class Cube "_box" ，事實上，就是 "_box" 繼承了 Cube 這個內建的class。

可以想像 Cube 如果寫成Class的話可以示意成這樣

當然，實際上不只這麼簡單，不只是 Cube 應該有更多的attribute，同時也會制定如何解讀這些attribute，比如 size 變為10時，Cube的geometry該做甚麼樣的變化。

事實上，Schema分成兩大類，IsA與API， Cube 就是屬於前者IsA Schema，

IsA Schema

也被稱之為Typed schema or Prim schema，意思是說這種schema是用來定義一個Prim的Type的，更進一步來說， Cube 是屬於可以 Concrete 的IsA Schema，也就是可以直接用 Cube 這個 typeName 去實例化prim，反之， Abstract(non-concrete) 的IsA Schema則常是內部開發用的基底，雖然有 typeName 但不能直接拿來實例化，例如: UsdGeomImageable 。

API Schema

相對於IsA，API Schema則不是用來實例化的，所以自然都是Abstract也沒有 typeName ，就像是prim使用IsA Schema制定格式，那在功能的格式制定則是使用API Schema，API Schema是開放給不同開發者使用的統一規範，IsA Schema則更像是創作者與創作者之間asset使用的規範。

API Schema只需要制定好API，因此可能缺乏實作，需要由真正的開發人員去依據自己的需求實作，因此這部分的格式制定偏向開發、coding，不太容易在Omniverse介面上看出太多端倪，所以這個部分暫且跳過。

總而言之，Schema就是格式的制定，或者說是data modeling，無論是prim或是function，我們可以進一步思考一個有趣的問題，IsA Schema與class specifier差異在哪，因為我剛剛有提到

你完全可以想像成Schema是一個內建的class

首先，兩者都能夠制定prim的template，但是後者沒辦法進一步將這些資訊依據事先寫好的規則影響其他的data。

可以想像成 Schema 是prim最基本的type，事先定義清楚這種prim的所有格式以及行為，包含這個prim長甚麼樣子，例如Cube的幾何由8個vertices構成，rendering的時候則是12個triangle之類的。

Class 則是基於這些基本的prim所做的擴充，例如這個Cube的顏色。當然，顏色也可以被規範在IsA Schema，設計Cube的開發者需要去考慮顏色是否是Cube必要的Attribute。相對而言，Class specifier的優點在於相當易用，但也僅能夠custom一些簡單的attribute，但是這些attribute該怎麼樣利用則需要後續開發。

但是，Schema不是必需的，例如 def "Box" 是合法的，它定義出最簡單的prim，反映到Omniverse的Property就只有Raw USD Properties，如Figure 2.a。但是如 def Cube "Box" 或是 class Cube "_box" ，此時可以發現Property就有了其他的attribute，例如Geometry、Visual和Kind，這些格式的規範都是來自於 Cube 這個Schema，前者可以參考Figure 1(b)，後者則參考Figure 2(b)

另外，可以補充，在 Layout.usda 中有一段 defaultPrim = "World" ，這行是在如果發生這個檔案被reference或是preload的時候，引用者的entry point，關於這兩個機制就留待後面介紹
